# Configuration Guide

Configuration in `devportal-platform` is preset-first. Most operators
will spend more time choosing presets and setting env vars than
hand-editing `app-config.yaml`. This doc covers both: the preset
surface (what most deployments touch) and the raw Backstage surface
(when a preset doesn't cover your case).

The canonical references are:

- [`presets/README.md`](../presets/README.md) — preset model, tiers,
  curation boundary.
- [`presets/SCHEMA.md`](../presets/SCHEMA.md) — preset file format.
- [`entrypoint.sh`](../entrypoint.sh) — the runtime that assembles
  everything at boot.

This guide tells you when to reach for each.

## The two paths

The image supports two equally-first-class operator paths
([`presets/README.md`](../presets/README.md) § "Two primary paths of
use"):

| Path              | When to use                                                                                    | What you set                                                                                                     |
| ----------------- | ---------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| **Preset**        | Your stack matches a catalog entry (GitHub, Keycloak, Azure, …).                               | `VEECODE_PRESETS=recommended,<integration>,…` + the env vars each preset declares as required.                   |
| **Raw Backstage** | You need something a preset doesn't cover, or you have a Helm chart that produces these files. | Mount `app-config.local.yaml` and, when needed, a `dynamic-plugins.yaml` with top-level `plugins:` entries. Leave `VEECODE_PRESETS` unset. |

The two paths layer — even a preset deployment can drop in an
`app-config.local.yaml` for overrides, and that file always wins.

## The config chain at boot

`entrypoint.sh` assembles the backend's `--config` flags in this
order. Later files override earlier ones for overlapping keys
(Backstage's config loader deep-merges objects and last-write-wins on
scalars):

```
1. app-config.yaml                       (base distribution defaults)
2. app-config.production.yaml            (container/production overrides)
3. app-config.distro.yaml                (~10-line distro defaults)
4. app-config.preset-<name>.yaml          (one per selected preset, in VEECODE_PRESETS order)
5. app-config.local.yaml                  (operator overrides via volume or VEECODE_APP_CONFIG)
6. dynamic-plugins-root/app-config.dynamic-plugins.yaml  (generated by install-dynamic-plugins.py)
7. app-config.saas.yaml                   (SaaS overrides — VEECODE_APP_CONFIG, decoded from base64)
```

A few practical consequences:

- **Your `app-config.local.yaml` wins over preset config.** That's
  intentional — operators can always trim or override what a preset
  set without forking the preset.
- **Dynamic plugin configs land late.** A plugin's `pluginConfig:`
  ends up in `app-config.dynamic-plugins.yaml`, after the local
  overrides. So a plugin that ships an opinionated config can step on
  your local config. If that happens, the fix is to either (a) move
  the override into a preset's `appConfig:` (which loads earlier and
  is plugin-aware), or (b) re-declare the override in
  `app-config.saas.yaml` if you control SaaS injection.
- **`app-config.saas.yaml` always wins.** Reserved for SaaS-time
  overrides (database URLs, etc.). Don't use it in normal deployments.

For raw Backstage path: skip the `app-config.preset-<name>.yaml`
layer (it doesn't get written if `VEECODE_PRESETS` is unset). The
rest of the chain still applies.

## Preset path

A preset declares three things:

```yaml
name: github
description: GitHub-integrated IDP stack
version: 1.0.0

requires:
  variables:
    GITHUB_PAT:
      description: Personal Access Token (scope repo + read:org)
      required: true
      docs: https://docs.github.com/…

plugins:
  - package: oci://${PLUGIN_REGISTRY}/github-actions:bs_1.48.4!backstage-community-plugin-github-actions
    disabled: false
  - package: oci://${PLUGIN_REGISTRY}/tech-radar:bs_1.49.4!backstage-community-plugin-tech-radar
    disabled: false

appConfig:
  integrations:
    github:
      - host: github.com
        token: ${GITHUB_PAT}
```

What happens at boot, per preset:

1. **Required variables** are checked. Any missing `required: true`
   variable fails the boot with exit 78 and a list of every missing
   var across every selected preset.
2. **`plugins`** are written into `/app/preset-<name>-plugins.yaml`
   and added to the list of plugin files loaded at boot. The install
   script then loads each one and merges per `package:` key —
   last-write-wins on the whole entry.
3. **`appConfig`** is written into
   `/app/app-config.preset-<name>.yaml` and added to the backend's
   `--config` chain (in the preset's position in the list).

Preset order matters when two presets set the same config key. Later
presets win on scalar values; objects deep-merge. If `recommended`
sets `app.title: 'VeeCode DevPortal'` and `veecode-theme` overrides
it to `app.title: 'VeeCode'`, the order in `VEECODE_PRESETS` decides
which one ends up active.

### Adding a new preset

See [`presets/README.md`](../presets/README.md) § "Adding a new
preset" + [`presets/SCHEMA.md`](../presets/SCHEMA.md). The short
form:

1. Copy `recommended.yaml` as a starting point.
2. Pick a unique `name` and one-line `description`. Bump `version`
   per semver rules in the schema doc.
3. List the plugins your stack needs in `plugins:` — prefer OCI refs
   over npm.
4. Declare every env var your plugins need in `requires.variables`
   (with `description`, `required`, `docs` URL, optional `example`).
5. Provide the `appConfig:` that wires plugins to those vars.
6. Test locally: `VEECODE_PRESETS=<your-preset> ./scripts/dev-run.sh run`.

A preset is a contract between the image and the operator, not a
config snippet. The contract says: "to enable this stack, set X, Y,
Z; here are docs". Treat the curation boundary in
[`presets/README.md`](../presets/README.md) § "The curation boundary"
as load-bearing.

## Raw Backstage path

Two things to know.

### `app-config.local.yaml` (or volume-mounted equivalent)

This is your override file. It's read after every preset's config and
overrides anything. Typical contents for a deployment that wants its
own auth wiring:

```yaml
auth:
  environment: production
  providers:
    github:
      production:
        clientId: ${AUTH_GITHUB_CLIENT_ID}
        clientSecret: ${AUTH_GITHUB_CLIENT_SECRET}

integrations:
  github:
    - host: github.com
      token: ${GITHUB_PAT}

catalog:
  providers:
    github:
      organization: ${GITHUB_ORG}
      schedule:
        frequency: { minutes: 30 }
        timeout: { minutes: 3 }
```

Mount it at `/app/app-config.local.yaml`:

```bash
docker run -p 7007:7007 \
  -v $(pwd)/app-config.local.yaml:/app/app-config.local.yaml:ro \
  veecode/devportal-platform:latest
```

Or set `VEECODE_APP_CONFIG` to its base64-encoded contents — the
entrypoint decodes it into `app-config.saas.yaml`
([`entrypoint.sh`](../entrypoint.sh)).

### `dynamic-plugins.yaml`

This file enables plugins by hand, separately from presets. Add entries
to its top-level `plugins:` list; the catalog, marketplace, and preset
plugins are assembled automatically at boot, so this file carries only
your own entries. If you want to add your own plugins outside of
a preset, the cleanest path is to volume-mount an `app-config.local.yaml`
that sets the `dynamicPlugins.frontend.<name>` config, and either:

- Drop a built plugin into `/app/dynamic-plugins-root/<name>/` via
  volume (no install step needed — the bundle is just present).
- Or mount a custom `dynamic-plugins.yaml` containing `plugins:` entries
  and let the install script download/install those entries.

The preset path covers the common cases more cleanly, so this should
be the exception.

## App-config variable substitution

`${VAR_NAME}` is expanded against the process environment at startup
(Backstage native). Unset vars become empty strings unless declared
as `required: true` in a preset's `requires.variables` (in which case
the preset fails the boot first).

`${VAR_NAME:-default}` is NOT supported — that's shell syntax, not
Backstage syntax. The closest Backstage offers is field-level
optionality (a missing var in an optional field just doesn't apply).

## Reserved env-var prefixes

[`presets/SCHEMA.md`](../presets/SCHEMA.md) § "Reserved env var
prefixes":

| Prefix                                                  | Owner                                                                            |
| ------------------------------------------------------- | -------------------------------------------------------------------------------- |
| `VEECODE_*`                                             | platform (image-wide; `VEECODE_PRESETS`, `VEECODE_APP_CONFIG`, `VEECODE_DOMAIN`) |
| `BACKSTAGE_*`                                           | upstream Backstage core (e.g. `BACKSTAGE_VERSION`)                               |
| `<PROVIDER>_*` (`GITHUB_*`, `AZURE_*`, `KEYCLOAK_*`, …) | preset-specific                                                                  |
| `AUTH_*`                                                | OAuth client IDs/secrets                                                         |
| `LOG_LEVEL`, `NODE_OPTIONS`, `DEBUG_PORT`               | runtime                                                                          |

A preset declaring a variable outside its provider namespace is a
smell — either the variable is platform-wide (don't put it in the
preset) or the preset bundles two integrations (split it).

## Common config keys

For reference, the most-touched keys in
[`app-config.yaml`](../app-config.yaml):

- `app.title` / `app.baseUrl` — title shown in the header; URL the
  browser uses. In production set both via `app-config.local.yaml`
  or a preset.
- `app.branding.{fullLogo,iconLogo,fullLogoWidth,theme}` — logos and
  palette. The `veecode-theme` preset overrides this; raw deployments
  can set their own.
- `backend.baseUrl` — URL the browser uses to reach the backend.
  Default `:7007` matches the image's bound port; production
  deployments typically front this with a reverse proxy.
- `backend.csp` / `backend.cors` — Helmet CSP directives and CORS
  whitelist. Default policy is restrictive; add hosts to `img-src`
  for external avatars/badges, `connect-src` for backend-proxied
  APIs.
- `backend.database` — defaults to in-memory SQLite (dev). For
  production, set `client: pg` and a connection. The image bakes in
  the `pg` driver.
- `auth.environment` — `development` or `production`. The shipped
  `app-config.yaml` is `development`; an integration preset sets it
  to `production` if appropriate.
- `auth.providers.<name>` — wiring per provider. Presets fill these
  out; raw deployments fill them out themselves.
- `catalog.locations` — what to ingest. The image ships
  `examples/catalog-all.yaml` in the base config and switches to
  `examples/*.yaml` in `app-config.production.yaml`.
- `permission.rbac.policies-csv-file` — RBAC policy path. Defaults to
  `${RBAC_POLICY_PATH:-/app/rbac-policy.csv}`; see [`RBAC.md`](RBAC.md).
- `techdocs.{builder,publisher,generator}` — TechDocs storage and
  rendering. Defaults to `local` everywhere; for production, switch
  to `googleGcs`/`awsS3`/`azureBlobStorage` per Backstage docs.
- `dynamicPlugins.rootDirectory` — `dynamic-plugins-root`. Don't
  change.

## What NOT to put in config

- **Secrets in `app-config.yaml`** (committed). Use env vars and
  `${VAR}` substitution.
- **Business logic in presets** ([`presets/README.md`](../presets/README.md)
  § "Discipline"). Presets are configuration scaffolds, not feature
  flags.
- **Customer-specific RBAC** in `rbac-policy.csv` shipped in the
  image. Mount a customer-specific CSV (`RBAC_POLICY_PATH`) at deploy
  time. See [`RBAC.md`](RBAC.md).
- **Anything that would be wrong if shared across customers** —
  catalog rules that assume an org structure, scaffolder templates
  that hard-code your registry. Customer artifacts go in customer
  deployments.

## Troubleshooting

**`ERROR: preset "<name>" not found`** — typo in `VEECODE_PRESETS`,
or you renamed the file but the env var still points at the old
name. `ls /app/presets/` (or `ls presets/` in the repo) for the
canonical list.

**`Preset "github" requires GITHUB_PAT`** — exit code 78. The env
var is missing or empty. Set it before boot.

**A preset's plugin is installed twice** — the `package:` field in
the preset doesn't match the entry in `dynamic-plugins.default.yaml`
exactly. Check for trailing `-dynamic` suffixes
([`presets/SCHEMA.md`](../presets/SCHEMA.md) § "Composition" calls
out `…-backend-dynamic-dynamic`).

**Your local override didn't take effect** — config layer order
matters. If a dynamic plugin's `pluginConfig:` overrode you, move
the override to a preset's `appConfig:` (so it loads with the
preset, before the dynamic-plugins layer) or to
`app-config.saas.yaml` (which wins last).

**Catalog locations from the shipped `examples/` aren't appearing** —
that data is only loaded in container runs (`app-config.production.yaml`
points at `/app/examples/`). For `yarn dev-local`, the locations
in `app-config.yaml` use a relative path that resolves from
`packages/backend`.
